# План API-тестирования QA Lab

Документ описывает будущий набор автотестов для CRUD API дефектов. Это чек-лист покрытия, а не реализация тестов. Источник истины по контракту — [`openapi.yaml`](./openapi.yaml), детали архитектуры — [`api-lab.md`](./api-lab.md).

## Цели

- доказать корректность полного CRUD-цикла;
- проверить валидацию тела, path/query-параметров и обязательного workspace-заголовка;
- подтвердить изоляцию демо-пространств, лимит записей и отсутствие потерянных обновлений;
- контролировать стабильность схем, статусов и служебных заголовков;
- получать независимые и воспроизводимые результаты локально и в CI.

## Подход и окружение

Предпочтительный стек — Playwright `APIRequestContext`, чтобы использовать уже установленный test runner, HTML-отчёт, trace и общую CI-конфигурацию. API-тесты стоит вынести в отдельный каталог, например `tests/api`, и запускать отдельным npm-скриптом.

Для каждого теста или изолированного блока создаётся новый UUID в `X-Demo-Workspace-Id`. Это исключает зависимость от порядка запуска и параллельных worker-ов. Данные production-сайта в автотестах не используются.

Базовые правила фикстур:

- генерировать workspace UUID средствами Node.js перед тестом;
- иметь фабрику валидного payload с возможностью переопределить отдельные поля;
- сохранять созданные ID только внутри теста;
- не полагаться на очистку общего workspace;
- проверять `Content-Type`, `Cache-Control`, `Vary` и соответствие `X-Request-Id` полю `error.requestId`;
- при падении прикладывать request/response без чувствительных данных;
- запускать каждый сценарий независимо и допускать произвольный порядок выполнения.

## Приоритеты

- **P0** — основной CRUD и доступность контракта;
- **P1** — валидация, фильтры, изоляция и лимиты;
- **P2** — конкурентность, срок хранения и расширенные проверки контракта.

## Чек-лист

### P0 — основной CRUD

- [ ] `POST /api/issues` с обязательными полями возвращает `201`, созданный объект и `Location: /api/issues/{id}`.
- [ ] При создании без `status` сервер устанавливает `open`.
- [ ] Сервер обрезает пробелы по краям `title` и `description`.
- [ ] `id` имеет формат UUID; `createdAt` и `updatedAt` имеют формат ISO 8601 и при создании равны.
- [ ] `GET /api/issues/{id}` возвращает созданный объект с `200`.
- [ ] `GET /api/issues` содержит созданный объект и корректный `total`.
- [ ] `PATCH /api/issues/{id}` изменяет только переданные поля, сохраняет `id` и `createdAt`, увеличивает `updatedAt`.
- [ ] `DELETE /api/issues/{id}` возвращает `204` с пустым телом.
- [ ] После удаления `GET /api/issues/{id}` возвращает `404 NOT_FOUND`, а объект отсутствует в списке.

### P1 — создание и валидация тела

- [ ] Границы `title`: 3 и 120 Unicode-символов принимаются; 2 и 121 отклоняются с `422` и ошибкой поля.
- [ ] Границы `description`: 10 и 2000 Unicode-символов принимаются; 9 и 2001 отклоняются.
- [ ] Строки из пробелов проверяются после `trim` и отклоняются.
- [ ] Каждое значение `severity` (`low`, `medium`, `high`, `critical`, `blocker`) принимается.
- [ ] POST принимает каждое начальное значение `status` (`open`, `in_progress`, `testing`, `resolved`).
- [ ] Неизвестные enum-значения, другой регистр и неверный тип возвращают `422 VALIDATION_ERROR`.
- [ ] Отсутствие каждого обязательного поля POST возвращает `422` и ключ поля в `error.fields`.
- [ ] JSON `null`, массив, строка и число вместо объекта возвращают `422` с ключом `_body`.
- [ ] Неизвестное поле возвращает `422`; защищённые `id`, `createdAt`, `updatedAt` возвращают отдельные ошибки полей.
- [ ] Пустой объект PATCH возвращает `422`, а PATCH с одним допустимым полем проходит.
- [ ] PATCH не обнуляет не переданные поля.
- [ ] Некорректный JSON возвращает `400 MALFORMED_JSON`.
- [ ] Отсутствующий, пустой или отличный от `application/json` Content-Type для POST/PATCH возвращает `415 UNSUPPORTED_MEDIA_TYPE`.
- [ ] `application/json; charset=utf-8` принимается.

### P1 — статусная модель

- [ ] Допустимы переходы вперёд `open → in_progress`, `in_progress → testing`, `testing → resolved`, а также возвраты `testing → in_progress` и `resolved → open`.
- [ ] Повторная передача текущего статуса допустима для всех четырёх состояний.
- [ ] Все остальные переходы с пропуском шага или возвратом назад возвращают `409 INVALID_STATUS_TRANSITION` с ключом `status` в `error.fields`.
- [ ] PATCH без `status` продолжает изменять остальные поля в любом состоянии.
- [ ] Запрещённый смешанный PATCH не изменяет ни одно поле и не увеличивает `updatedAt`.
- [ ] Проверка выполняется атомарно по фактическому текущему статусу записи.

### P1 — workspace и идентификаторы

- [ ] Отсутствующий, пустой и невалидный `X-Demo-Workspace-Id` возвращает `400 INVALID_WORKSPACE`.
- [ ] UUID в верхнем регистре принимается.
- [ ] Невалидный `{id}` для item-маршрута возвращает `400 INVALID_ID`.
- [ ] Валидный, но отсутствующий `{id}` возвращает `404 NOT_FOUND` для GET/PATCH/DELETE.
- [ ] Два workspace могут создать разные записи и видят только собственный список.
- [ ] Один и тот же issue ID нельзя получить из другого workspace.
- [ ] Очистка клиентского `localStorage` не является API-операцией и не должна влиять на данные другого UUID.

### P1 — список, поиск и фильтры

- [ ] Пустой workspace возвращает `{ "items": [], "total": 0 }`.
- [ ] Список отсортирован по `updatedAt` по убыванию; обновлённая запись перемещается вверх.
- [ ] Один `status` возвращает только точное значение статуса.
- [ ] Один `severity` возвращает только точное значение важности.
- [ ] Повторённые `status` и `severity` объединяют значения внутри поля по OR.
- [ ] `q` ищет подстроку одновременно в title и description без учёта регистра.
- [ ] Пробелы по краям `q` игнорируются; пустой после `trim` запрос эквивалентен отсутствию поиска.
- [ ] `q` длиной 120 символов принимается, длиной 121 — `400 INVALID_QUERY`.
- [ ] Фильтры `q`, `status` и `severity` комбинируются между полями по логике AND.
- [ ] `total` равен длине отфильтрованного массива, а не общему числу workspace.
- [ ] Невалидные значения status/severity возвращают `400 INVALID_QUERY` и ошибку соответствующего поля.
- [ ] Неизвестный query-параметр возвращает `400 INVALID_QUERY`.
- [ ] Повторённый `q` возвращает `400 INVALID_QUERY`.
- [ ] Точные дубликаты `status` или `severity` возвращают `400 INVALID_QUERY`.

### P1 — лимит и методы

- [ ] Первые 50 POST-запросов в одном workspace успешны.
- [ ] 51-й POST возвращает `409 WORKSPACE_LIMIT_REACHED`; число записей остаётся 50.
- [ ] После удаления одной записи создание снова разрешено.
- [ ] Неподдерживаемый метод коллекции возвращает `405 METHOD_NOT_ALLOWED` и `Allow: GET, POST`.
- [ ] Неподдерживаемый метод item-ресурса возвращает `405 METHOD_NOT_ALLOWED` и `Allow: GET, PATCH, DELETE`.
- [ ] Проверка метода выполняется до workspace-валидации: неподдерживаемый метод без заголовка всё равно возвращает `405`.

### P1 — единый формат ответа

- [ ] Каждый JSON-ответ имеет `Content-Type: application/json; charset=utf-8`.
- [ ] Каждый ответ, включая `204`, содержит `Cache-Control: no-store`, `Vary: X-Demo-Workspace-Id` и UUID в `X-Request-Id`.
- [ ] Любая ошибка имеет объект `error` с `code`, `message`, `fields` и `requestId`.
- [ ] `error.requestId` совпадает с заголовком `X-Request-Id`.
- [ ] Успешный DELETE не содержит тело и Content-Type.

### P2 — конкурентность и целостность

> Эти сценарии запускаются на Netlify deploy preview. Локальный Blobs-сервер
> `netlify-cli 26.2` не гарантирует атомарность одновременных conditional writes;
> подробности зафиксированы в `api-lab.md`.

- [ ] Параллельные POST в пустой workspace не теряют успешные записи.
- [ ] При параллельном заполнении workspace итоговое число записей не превышает 50.
- [ ] Итоговый список содержит ровно объекты всех успешных POST; ответы `409` допустимы только с документированным кодом.
- [ ] Параллельные PATCH разных записей сохраняют оба изменения.
- [ ] Параллельные PATCH одной записи не повреждают схему документа; каждый успешный ответ можно затем прочитать как валидный Issue.
- [ ] После исчерпания ETag retry API возвращает `409 CONCURRENT_MODIFICATION` в едином формате.
- [ ] `updatedAt` строго увеличивается даже для двух быстрых последовательных PATCH одной записи.

### P2 — срок активности workspace

- [ ] Workspace до даты истечения читается как обычно.
- [ ] Просроченный workspace логически возвращается как пустой.
- [ ] GET не продлевает срок активности.
- [ ] Успешные POST, PATCH и DELETE устанавливают срок ещё на 30 дней.
- [ ] Первая мутация после истечения перезаписывает просроченный документ и не восстанавливает старые записи.

Эти проверки лучше выполнить на уровне интеграционных тестов хранилища с управляемыми часами и тестовым адаптером Blobs: ожидание 30 дней в E2E недопустимо. Физическое удаление Blob не входит в текущий контракт.

### P2 — OpenAPI и обратная совместимость

- [ ] `docs/openapi.yaml` проходит синтаксическую и OpenAPI 3.1-валидацию.
- [ ] Для каждого operation зафиксированы обязательный workspace header, path/query-параметры, request body и все фактические статусы.
- [ ] Успешные ответы и ошибки валидируются против схем OpenAPI.
- [ ] Реальные enum-значения и ограничения длины совпадают с контрактом.
- [ ] Изменение или удаление поля/статуса требует осознанного обновления контракта и тестов.

## Нефункциональные проверки

- [ ] Время ответа для CRUD-сценария записывается как метрика без жёсткого SLA для serverless cold start.
- [ ] Ответы не содержат stack trace, внутренние ключи Blob store или абсолютные пути.
- [ ] В серверном логе ошибка может быть найдена по `X-Request-Id`.
- [ ] Payload с HTML/JavaScript остаётся обычным текстом в API и безопасно отображается UI через `textContent`.
- [ ] Максимальные допустимые payload и список из 50 элементов не приводят к ошибке размера ответа.

## Критерии готовности набора тестов

- все P0- и P1-сценарии автоматизированы и стабильно проходят локально и в CI;
- тесты не используют общий статический workspace и выдерживают параллельный запуск;
- OpenAPI-контракт валидируется автоматически;
- flaky retry не скрывает продуктовые ошибки конкурентности;
- отчёт позволяет восстановить запрос, статус и безопасную часть ответа по каждому падению.
