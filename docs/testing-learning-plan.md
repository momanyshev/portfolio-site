# План исследования sim-use и проверки гипотезы Agentic Mobile QA

Этот документ — рабочий чек-лист исследования `sim-use` на мобильных
приложениях iOS и Android. Основным тренажёром служит QA Lab Mobile
(`ru.maksim.qalab`), а завершающая проверка переносимости проводится на втором,
независимом мобильном приложении.

Документ не является планом создания обычного смоук- или регрессионного набора.
Его цель — экспериментально определить возможности, ограничения, надёжность и
практическую ценность `sim-use` как инструмента, через который AI-агент наблюдает
мобильный интерфейс и действует в нём.

## Цель и ожидаемый результат

Проверяемая гипотеза:

> AI-агент с помощью `sim-use` способен выполнять содержательные QA-задачи в
> мобильном приложении, независимо подтверждать результат через UI и backend,
> собирать воспроизводимые доказательства и сокращать активное ручное управление
> iOS Simulator или Android Emulator.

После прохождения плана должны появиться:

- обоснованное мнение о реальных возможностях и границах `sim-use`;
- практическое знание команд, селекторов, платформенных различий и способов
  восстановления после ошибок;
- набор воспроизводимых исследовательских сценариев и измерений;
- отдельные выводы для iOS и Android;
- проверка переносимости опыта на второе мобильное приложение;
- собственный AI-агент, в котором закреплены полученные правила, runbook,
  API-оракулы, ограничения, схема отчёта и обработка ошибок.

## Границы исследования

В исследование входят:

- нативное мобильное приложение QA Lab на iOS Simulator;
- нативное мобильное приложение QA Lab на Android Emulator;
- при необходимости дополнительная проверка на выделенном Android-устройстве;
- взаимодействие мобильного клиента с REST API `/api/issues`;
- API как источник тестовых данных и независимый оракул результата;
- accessibility tree, ввод, жесты, системные состояния, жизненный цикл
  приложения, скриншоты, видео и crash awareness;
- guided, goal-based и exploratory задания для AI-агента;
- второе мобильное приложение с собственным backend и контролируемыми данными;
- создание собственного AI-агента поверх `sim-use`.

В исследование не входят:

- замена всех существующих уровней тестирования одним AI-агентом;
- проверка производительности, нагрузки или безопасности API средствами
  `sim-use`;
- автоматическая публикация найденных дефектов без человеческого ревью;
- использование production-данных, production-аккаунтов или чувствительной
  информации;
- объединённый прогон iOS и Android с одним общим verdict: платформы всегда
  проверяются и оцениваются раздельно.

## Как работать с планом

- Менять `[ ]` на `[x]` только после наблюдаемого действия или появления
  указанного артефакта.
- Если пункт неприменим, отмечать его как `N/A` и рядом записывать причину.
  Этапы 12 и 14, а также их обязательные iOS/Android gates нельзя пропускать или
  помечать `N/A`: без второго приложения и собственного агента план не завершён.
- Не переходить к следующему этапу, пока не выполнен критерий завершения
  текущего.
- Для каждого эксперимента сохранять исходное состояние, инструкцию агенту,
  transcript, конечное состояние, API before/after и итоговый verdict.
- Не считать фразу агента «задача выполнена» доказательством успеха. `PASS`
  выставляет только независимый оракул.
- Не объединять метрики iOS и Android до получения отдельных результатов по
  каждой платформе.

Названия команд, полей, файлов и форматов сохраняются на английском, потому что

Расшифровки ниже объясняют значение терминов в контексте этого плана.

### HTTP и API

| Термин | Перевод | Значение в плане |
| --- | --- | --- |
| **Backend** | Серверная часть приложения | Принимает API-запросы, применяет бизнес-правила и хранит данные |
| **Endpoint** | Конечная точка API | Конкретный URL и HTTP-метод, например `GET /api/issues` |
| **Request** | Запрос | Данные, отправленные мобильным клиентом в backend |
| **Response** | Ответ | Статус, заголовки и данные, которые backend вернул клиенту |
| **Request/response body** | Тело запроса/ответа | JSON-данные внутри HTTP-сообщения |
| **Payload** | Полезная нагрузка | Передаваемые данные запроса, ответа или test fixture |
| **Query** | Параметры запроса | Часть URL для поиска, фильтрации и других параметров чтения |
| **Header** | Заголовок HTTP | Метаданные запроса или ответа, например Workspace ID и `requestId` |
| **Contract** | Контракт | Зафиксированные правила endpoint, полей, статусов и ошибок API |
| **GET** | Получить ресурс | Читает список или один дефект без намеренной мутации |
| **POST** | Создать ресурс | Создаёт новый дефект |
| **PATCH** | Частично изменить ресурс | Обновляет только переданные поля существующего дефекта |
| **DELETE** | Удалить ресурс | Удаляет дефект |
| **200 OK** | Успешный ответ | Запрос выполнен, ответ содержит данные |
| **201 Created** | Ресурс создан | Успешный результат `POST` |
| **204 No Content** | Успех без тела ответа | Успешный результат `DELETE` |
| **404 Not Found** | Ресурс не найден | Ожидаемый ответ при чтении удалённого или неизвестного дефекта |
| **500 Internal Server Error** | Внутренняя ошибка сервера | Искусственно вызываемый или реальный сбой backend |
| **requestId** | Идентификатор запроса | Позволяет связать API-ответ с логом и конкретной операцией |
| **API inspector** | Инспектор API | Экран QA Lab Mobile с деталями последней пользовательской HTTP-операции |
| **Timestamp** | Временная отметка | Зафиксированные дата и время события, запроса или artifact |
| **Workspace** | Изолированное демонстрационное пространство | Набор данных, отделённый заголовком `X-Demo-Workspace-Id` |
| **Test namespace** | Тестовое пространство имён | Общее название изолированной области данных; Workspace — его реализация в QA Lab |

### Сущности эксперимента

| Термин | Перевод | Значение в плане |
| --- | --- | --- |
| **Case** | Экспериментальный сценарий | Одна задача с инструкцией, начальным состоянием, oracle и ожидаемыми artifacts |
| **Run** | Отдельный запуск | Полное выполнение case из сброшенного состояния с новым Workspace; retry одного действия не является новым run |
| **Case manifest** | Манифест сценария | Структурированное описание case: platform, instruction, limits, oracle, evidence и teardown |
| **Version manifest** | Манифест версий | Список версий приложения, `sim-use`, skill, модели, OS и устройства конкретного run |
| **Precondition** | Предусловие | Состояние, которое должно существовать до запуска задания |
| **Postcondition** | Постусловие | Проверяемое состояние, которое должно существовать после выполнения задания |
| **Ground truth** | Эталонная истина | Заранее известный фактический результат, с которым сравнивается verdict |
| **Oracle** | Оракул, независимая проверка | Механизм, который определяет фактический успех через API, UI или другое детерминированное состояние |
| **Verdict** | Итоговый вердикт | Формальный результат run: `PASS`, `FAIL`, `BLOCKED` или `INCONCLUSIVE` |
| **Evidence** | Доказательство | Один подтверждающий artifact: outline, screenshot, API state, log и т. п. |
| **Evidence pack / bundle** | Комплект доказательств | Совокупность artifacts, достаточная для проверки verdict и воспроизведения результата |
| **Artifact** | Артефакт | Сохранённый файл или запись, полученные в ходе run |
| **Transcript** | Протокол взаимодействия | Полная последовательность инструкций, рассуждений, tool calls и результатов |
| **Log** | Журнал событий | Запись действий системы с временем, статусом и диагностическими данными |
| **Seed** | Подготовка начальных данных | Создание fixtures через API до выполнения UI-задачи |
| **Fixture** | Тестовая фикстура | Конкретная заранее подготовленная запись или набор данных |
| **Teardown** | Завершающая очистка | Удаление данных и сброс окружения после любого результата run |
| **Reset** | Сброс | Возврат приложения, proxy, устройства или данных в известное исходное состояние |
| **Clean control run** | Чистый контрольный запуск | Run без намеренного дефекта, используемый для измерения ложных находок |
| **Seeded defect** | Намеренно посеянный дефект | Контролируемая ошибка с заранее известным ground truth |
| **Baseline** | Базовый уровень | Исходные показатели ручного выполнения или текущей версии решения |
| **Pilot** | Пилот | Небольшая первая серия runs перед полным benchmark |
| **Benchmark** | Сравнительное измерение | Стандартизированная серия runs для расчёта надёжности, времени и стоимости |

### Статусы и решения

| Статус | Расшифровка | Когда использовать |
| --- | --- | --- |
| **PASS** | Успех | Все обязательные postconditions подтверждены oracle |
| **FAIL** | Неуспех | Задание или проверяемая функция завершились неверным результатом |
| **BLOCKED** | Заблокировано | Результат нельзя получить из-за окружения, инструмента или внешней зависимости |
| **INCONCLUSIVE** | Недостаточно данных | Собранных данных недостаточно для честного `PASS` или `FAIL` |
| **N/A** | Not Applicable — неприменимо | Пункт не относится к конкретной ситуации; обязательные этапы 12 и 14 так отмечать нельзя |
| **GO** | Продолжать | Гипотеза получила достаточное подтверждение для следующего этапа |
| **PIVOT** | Изменить направление | Сохранить подтверждённую часть идеи, но сузить или изменить scope |
| **STOP** | Остановить | Практическая ценность в проверяемой форме не подтверждена |

### Метрики

| Термин | Перевод / смысл |
| --- | --- |
| **Correct verdict rate** | Доля runs, в которых итоговый verdict совпал с ground truth |
| **pass@1** | Доля cases, успешно выполненных с первого независимого run |
| **pass@3** | Доля cases, для которых хотя бы один из первых трёх независимых runs успешен |
| **False PASS** | Ошибочный `PASS`, когда требуемое состояние фактически не достигнуто |
| **False positive** | Ложная находка: агент сообщает о дефекте в чистой сборке |
| **False negative / missed defect** | Пропуск: существующий seeded defect не был найден |
| **Recall** | Полнота обнаружения: найденные seeded defects / все посеянные дефекты |
| **Intervention rate** | Доля runs, в которых человеку пришлось управлять устройством после старта |
| **Evidence completeness** | Доля runs с полным обязательным evidence pack |
| **Repeatability** | Способность case давать одинаковый подтверждённый результат в повторных runs |
| **Coordinate-free rate** | Доля успешных runs без нажатий по абсолютным координатам |
| **Active human time** | Время, когда человек непосредственно выполняет действия, а не ожидает завершения агента |
| **Tool call** | Один вызов агентом внешнего инструмента или команды |
| **Selector mix** | Распределение selectors | Доли действий по alias, ID, label и координатам |
| **Tokens** | Единицы текста AI-модели | Объём входа и выхода модели; это не буквальное число слов или символов |
| **Cost** | Стоимость | Денежные затраты на модель и необходимую инфраструктуру |

### Термины sim-use и мобильного UI

| Термин | Перевод | Значение в плане |
| --- | --- | --- |
| **sim-use** | Название open-source CLI | Даёт AI-агенту возможность наблюдать и управлять iOS Simulator и Android emulator/device |
| **Observe → Act → Verify** | Наблюдать → действовать → проверять | Обязательный цикл: получить актуальный UI, выполнить действие, независимо подтвердить результат |
| **Preflight** | Предварительная проверка | Проверка установки, устройства, приложения, bridge, daemon и API до первого действия |
| **Accessibility** | Доступность интерфейса | Свойства UI, благодаря которым элементы доступны людям с ограничениями и инструментам автоматизации |
| **AccessibilityService** | Системный сервис доступности Android | Даёт Android bridge доступ к accessibility tree и действиям на устройстве |
| **Accessibility tree** | Дерево доступности | Иерархия доступных UI-элементов, которую читает `sim-use` |
| **Outline** | Компактное описание экрана | Сокращённое текстовое представление accessibility tree |
| **Entry** | Запись outline/JSON | Один наблюдаемый UI-элемент |
| **Role** | Роль элемента | Тип UI-элемента: button, textbox, header и т. п. |
| **Label** | Доступная подпись | Человекочитаемое имя элемента |
| **Value** | Значение элемента | Текущее содержимое или состояние элемента |
| **Frame** | Рамка элемента | Координаты и размеры элемента на экране |
| **Selector** | Способ выбора элемента | Правило, по которому `sim-use` находит цель действия |
| **Alias `@N`** | Временный псевдоним | Короткая ссылка на элемент из последнего полного outline |
| **Identifier `#<id>`** | Стабильный идентификатор | Accessibility ID элемента; обычно устойчивее временного alias |
| **testID** | Тестовый идентификатор | Свойство React Native, которое может стать accessibility identifier на устройстве |
| **Stale alias** | Устаревший псевдоним | Alias от старого состояния UI, который нельзя безопасно использовать после изменения экрана |
| **Selector collision** | Коллизия selector | Несколько элементов соответствуют одному правилу поиска |
| **Multiple matches** | Несколько совпадений | Ошибка, когда selector не определяет единственную цель |
| **Regex / contains** | Регулярное выражение / поиск фрагмента | Способы найти элемент по шаблону или части label, если точное совпадение невозможно |
| **Coordinate fallback** | Координатный запасной способ | Нажатие по точке экрана, допустимое только после неуспеха accessibility selectors |
| **Tap / scroll / gesture** | Нажатие / прокрутка / жест | Основные способы взаимодействия с мобильным UI |
| **List cell** | Элемент списка | Отдельная строка или карточка внутри прокручиваемого списка |
| **Blind spot** | Слепая зона | Важный UI-элемент или состояние, отсутствующие в доступном представлении |

### Сетевые сбои и диагностическая инфраструктура

| Термин | Перевод | Значение в плане |
| --- | --- | --- |
| **Proxy** | Прокси, посредник | Слой между mobile app и backend, который наблюдает или изменяет трафик |
| **Reverse proxy** | Обратный прокси | Сервер-посредник, принимающий запрос приложения и перенаправляющий их в backend |
| **Pass-through** | Сквозной режим | Proxy только пересылает запросы и не изменяет ответы |
| **Test double** | Тестовый заменитель | Управляемая замена реального компонента для воспроизведения заданного поведения |
| **Fault injection** | Внедрение сбоев | Контролируемое создание ошибок, задержек и нестандартного порядка ответов |
| **Fault profile** | Профиль сбоя | Сохранённая конфигурация конкретного сетевого поведения |
| **Fail-first** | Сначала ошибка | Первый запрос завершается сбоем, последующий обрабатывается успешно |
| **Delay / delayed response** | Задержка ответа | Искусственно увеличенное время ответа backend |
| **Request log/counter** | Журнал/счётчик запросов | Доказывает точное количество, порядок и параметры HTTP-запросов |
| **Disconnect** | Разрыв соединения | Имитация временной недоступности сети или backend |
| **Malformed JSON** | Некорректный JSON | Ответ, нарушающий синтаксис ожидаемого формата данных |
| **Out-of-order responses** | Ответы не в порядке запуска | Более поздний запрос завершается раньше предыдущего и проверяет защиту от stale state |
| **Feature flag** | Переключатель функции | Настройка, включающая отдельное поведение или seeded defect без постоянной смены кода |

### Термины AI-агента и собственного инструмента

| Термин | Перевод | Значение в плане |
| --- | --- | --- |

| **Skill** | Навык агента | Пакет инструкций и вспомогательных материалов для специализированной работы |

| **Guided** | Управляемый режим | Агент получает бизнес-шаги, но сам выбирает команды и selectors |

| **Escalation** | Эскалация | Передача проблемы человеку с собранными доказательствами |
| **Destructive action** | Разрушающее действие | Удаление или изменение данных/окружения, которое трудно восстановить |

| **Verifier** | Проверяющий компонент | Код, который применяет oracle и выставляет формальный verdict |
| **Adapter** | Адаптер | Слой, связывающий generic agent logic с конкретным приложением или API |

## Этап 2. Модель observe → act → verify

### Что нужно понять

`sim-use` выполняет действие, но не доказывает бизнес-результат автоматически.
Каждая операция строится как отдельный цикл:

1. `observe` — получить актуальный `ui` outline;
2. `act` — выбрать элемент и выполнить одно осмысленное действие;
3. `verify` — заново прочитать UI и проверить независимое постусловие.

### Чек-лист

- [ ] Получить текстовый outline одного экрана.
- [ ] Получить тот же экран с `--json`.
- [ ] Найти в JSON platform, screen, entries, frames, roles, labels и IDs.
- [ ] Сравнить объём и удобство текстового outline и полного JSON.
- [ ] Выполнить действие по свежему alias `@N`.
- [ ] После навигации доказать, что старый alias нельзя считать актуальным.
- [ ] Выполнить действие по стабильному `#<id>`.
- [ ] Выполнить действие по точному label.
- [ ] Создать неоднозначный label-запрос и изучить ошибку multiple matches.
- [ ] Разрешить неоднозначность без координат с помощью более точного selector.
- [ ] Использовать координаты один раз как последний fallback и записать, почему
  accessibility selector оказался невозможен.
- [ ] После каждого действия повторно выполнить `ui`.
- [ ] Проверить, что успешный exit code действия не заменяет проверку результата.
- [ ] Изучить JSON error envelope и поля с диагностической подсказкой.

### Приоритет selectors

1. `@N` — сразу после свежего `ui`;
2. `#<id>` — для устойчивого обращения к известному accessibility identifier;
3. точный label или value;
4. regex/contains с обязательным ограничением области;
5. координаты — только как документированный последний fallback.

### Критерий завершения

- [ ] Выполнено действие без координат, а результат независимо подтверждён
  повторным `ui` и постусловием.
- [ ] Сформулирована собственная памятка по выбору selector.

## Этап 6. API fixtures и независимый oracle

REST API — естественная часть мобильного приложения. В исследовании он выполняет

1. создаёт точное начальное состояние;
2. независимо проверяет результат действий в мобильном UI;
3. удаляет тестовые данные после завершения.

Источник контракта: [`openapi.yaml`](./openapi.yaml).

### 6.1. Workspace и данные

- [ ] Создать helper для проверки точных полей и количества записей.
- [ ] Создать helper для удаления созданных записей.
- [ ] Проверить teardown после успешного эксперимента.
- [ ] Проверить teardown после `FAIL`, `BLOCKED` и аварийного завершения.

### 6.2. Правила oracle

- [ ] Для мутации сохранить API before и API after.
- [ ] Проверять не только изменённое поле, но и отсутствие нежелательных
  изменений остальных полей.
- [ ] Проверять отсутствие лишних записей через API, а отсутствие повторных
  запросов — через request log/counter fault proxy или тестовую backend-
  инструментацию. API before/after сам по себе не доказывает число запросов.
- [ ] Для Workspace isolation проверять оба пространства отдельными запросами.
- [ ] Для удаления проверять `404` отдельного ресурса и итоговый список.

- [ ] Сохранять `requestId`, HTTP-статус и длительность для диагностики.
- [ ] Выставлять `PASS` только после успешного oracle, независимо от формулировки

## Этап 8. Режимы работы AI-агента

Один успешный пошаговый сценарий не доказывает способность агента самостоятельно

Агент получает бизнес-шаги, но не команды `sim-use`, identifiers или

Агент получает только конечную цель и ограничения.

Агент получает область исследования, риски и лимит времени, но не ожидаемый

- [ ] Составить charter на Workspace и изоляцию данных.

- [ ] Составить charter на ошибки сети и восстановление.

- [ ] Выполнить каждый charter отдельно на Android.

### 10.4. Метрики

Независимый `run` — полный запуск case из сброшенного состояния с новым test
namespace. Повтор одного действия внутри run по retry policy не создаёт новый
run. `pass@3` использует максимум три независимых runs, а не три попытки tap.

| Метрика | Формула |
| --- | --- |
| Correct verdict rate | runs, где итоговый verdict совпал с ground truth / все evaluable runs |
| `pass@1` | cases, успешно завершённые в первом независимом run / все запущенные cases |
| `pass@3` | cases с хотя бы одним успешным результатом в первых трёх независимых runs / все запущенные cases |
| False PASS rate | runs с итоговым `PASS`, где ground truth не достигнут / все runs с итоговым `PASS` |
| Seeded defect recall | корректно найденные seeded-defect instances / все seeded-defect instances |
| False-positive rate | clean control runs с ошибочным product-defect verdict / все evaluable clean control runs |
| Intervention rate | runs с ручным управлением устройством после старта / все started runs |
| Evidence completeness | runs со всеми обязательными artifacts / все started runs |
| Repeatability 4/5 | cases с минимум четырьмя успешными runs из пяти / все cases полного benchmark |
| Coordinate-free rate | успешные runs без coordinate actions / все успешные runs |
| Manual time reduction | `(manual active time - agent active human time) / manual active time` |

`BLOCKED` из-за окружения считать отдельно и не исключать молча: в отчёте должны
быть как минимум `evaluable runs`, `blocked runs` и `all started runs`.

Дополнительно собирать полную длительность run, tool calls, action retries,
tokens, стоимость, selector mix, unintended mutations, неочищенные fixtures,
crashes, зависания, environment failures и разницу метрик iOS/Android.

### 10.5. Рекомендованные пороги MVP

| Метрика | Рекомендованный порог |
| --- | --- |
| Correct verdict | не менее 80% |
| Seeded defect recall | не менее 70% |
| False-positive defects | не более 10% |
| False PASS на критических сценариях | 0 |
| Evidence completeness | не менее 90% |
| Повторяемость | минимум 80% сценариев успешны в 4/5 runs |
| Снижение активного ручного времени | не менее 30% |
| Workspace leaks / unintended deletes | 0 |

### Pivot criteria

После двух циклов улучшения prompt, skill, fixtures и accessibility сузить область
применения до guided reproduction/evidence capture, если выполняется одно из
условий:

- correct verdict остаётся ниже 70%;
- seeded defect recall остаётся ниже 60%;
- false-positive defects превышают 15%;
- более 20% runs требуют ручного управления устройством;
- стоимость и время не дают преимуществ относительно ручного baseline;
- невозможно обеспечить безопасный reset и teardown.

### Критерий завершения

- [ ] Есть отдельная таблица метрик iOS.
- [ ] Есть отдельная таблица метрик Android.
- [ ] Есть объяснение разницы между платформами.
- [ ] Принято решение `GO`, `PIVOT` или `STOP` для следующего этапа.

## Этап 14. Создание собственного AI-агента

Финальный AI-агент создаётся только после завершения экспериментов. Он должен
закреплять доказанные практики, а не предположения начала исследования.

### 14.1. Контракт агента

Определить:

- вход: задача/charter, приложение, платформа, build, project adapter и policy;
- выход: структурированный verdict, evidence pack, классификация результата и
  draft дефекта при необходимости;
- разрешённые действия;
- запрещённые действия;
- retry budget;
- stop conditions;
- правила human approval;
- правила очистки данных;
- обязательные поля отчёта.

- [ ] Агент принимает только одну целевую платформу на run.
- [ ] Агент выполняет preflight до первого действия.
- [ ] Агент проверяет, что наблюдает нужное приложение.
- [ ] Агент следует циклу observe → act → verify.
- [ ] Агент использует зафиксированный приоритет selectors.
- [ ] Агент не считает exit code действия бизнес-подтверждением.
- [ ] Агент создаёт отдельный Workspace или test namespace.
- [ ] Агент использует сменный API oracle adapter.
- [ ] Агент сохраняет обязательные evidence.
- [ ] Агент останавливается при crash и сохраняет состояние.
- [ ] Агент эскалирует после исчерпания retry budget.
- [ ] Агент запрашивает подтверждение destructive action.
- [ ] Агент выполняет teardown даже после неуспеха.
- [ ] Агент формирует draft дефекта, но не публикует его автоматически.
- [ ] Агент различает product, environment, tool, accessibility, oracle и agent

## Приложение C. Taxonomy результатов

| Категория | Значение | Пример |
| --- | --- | --- |
| `product` | Дефект мобильного приложения или backend | Неверное состояние после PATCH |
| `test-data` | Ошибка seed, Workspace или teardown | Run получил чужую fixture |
| `environment` | Сбой Simulator, Emulator, Metro, API или сети | Устройство недоступно |
| `agent` | Ошибка рассуждения или выбора действия | Агент не проверил результат |
| `sim-use` | Ошибка CLI, daemon, bridge или selector resolution | Команда завершилась внутренней ошибкой |
| `accessibility` | Нужный UI невозможно надёжно наблюдать | Контрол скрыт canvas-wrapper |
| `oracle` | Неверная или недостаточная проверка постусловия | PASS выставлен по неполному API-check |
| `charter` | Задание двусмысленно или непроверяемо | Не определено ожидаемое состояние |

Для любого неуспешного run выбрать основную категорию и при необходимости одну
вторичную. Не исправлять prompt до классификации причины.
