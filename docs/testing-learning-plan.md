# План обучения Playwright и Playwright MCP

Цель: на примере `momanyshev.netlify.app` научиться писать устойчивые e2e-тесты, использовать Playwright MCP для исследования интерфейса и переносить найденные проверки в обычный регрессионный набор Playwright.

## 1. Карта приложения

Что изучаем:

- одностраничный сайт-визитка;
- якорная навигация;
- hero-блок;
- разделы `Обо мне`, `Опыт`, `Проекты`, `Контакты`;
- переключатель темы;
- внешние контактные ссылки;
- адаптивная верстка.

Результат: чеклист сценариев, которые можно автоматизировать.

## 2. База Playwright

Что изучаем:

- `@playwright/test`;
- `test`, `expect`, `page`;
- `baseURL`;
- браузерные проекты: Chromium, Firefox, WebKit;
- HTML-отчет.

Практика:

- проверить `title`;
- проверить главный заголовок `Максим`;
- проверить видимость основных разделов;
- запустить тесты во всех браузерах.

## 3. Устойчивые локаторы

Что изучаем:

- `getByRole`;
- `getByText`;
- `getByAltText`;
- accessible name;
- почему CSS-классы лучше не использовать как первый выбор.

Практика:

- найти ссылки навигации по роли;
- найти кнопку переключения темы по accessible name;
- проверить `alt` у изображения профиля.

## 4. Навигация

Что изучаем:

- клики по ссылкам;
- проверка якорных переходов;
- ожидания без ручных `timeout`.

Практика:

- кликнуть `Опыт`;
- кликнуть `Проекты`;
- кликнуть `Контакты`;
- проверить hero-кнопки `Опыт` и `Достижения и проекты`.

## 5. Тема и состояние

Что изучаем:

- `localStorage`;
- `data-theme`;
- `aria-pressed`;
- сохранение состояния после reload.

Практика:

- проверить начальную темную тему;
- переключить на светлую;
- проверить `localStorage["portfolio-theme"]`;
- перезагрузить страницу и убедиться, что тема сохранилась.

## 6. Контакты и внешние ссылки

Что изучаем:

- проверка `href`;
- тесты без реального ухода на внешние сайты;
- доступные имена ссылок.

Практика:

- проверить `mailto:m.manyshev@yandex.ru`;
- проверить Telegram;
- проверить LinkedIn;
- проверить GitHub.

## 7. Responsive

Что изучаем:

- `test.use({ viewport })`;
- mobile/desktop различия;
- горизонтальный скролл;
- адаптивное поведение навигации.

Практика:

- на desktop навигация видима;
- на mobile навигация скрыта;
- на узком экране нет горизонтального скролла;
- hero-кнопки корректно перестраиваются.

## 8. Debugging

Что изучаем:

- `--headed`;
- `--ui`;
- trace viewer;
- screenshots/videos on failure;
- HTML report.

Практика:

- намеренно сломать один assertion;
- открыть trace;
- понять, на каком шаге и почему тест упал.

## 9. Playwright MCP

Что изучаем:

- MCP как инструмент для исследования UI агентом;
- accessibility snapshot;
- поиск интерактивных элементов;
- разница между exploratory-проверкой и регрессионным тестом.

Практика:

- открыть сайт через Playwright MCP;
- попросить агента найти навигацию, кнопку темы и контакты;
- пройти сценарий смены темы;
- на основе наблюдений написать обычный Playwright-тест.

## 10. Рабочий цикл MCP + Playwright

Правило:

- Playwright MCP используется для исследования, быстрой проверки и диагностики.
- Playwright tests используются как источник правды и запускаются в CI.

Практика:

- MCP исследует сценарий;
- тест переносится в `tests/e2e`;
- тест запускается локально;
- результат фиксируется в репозитории.

## 11. CI

Что изучаем:

- запуск тестов на каждый push/PR;
- установка браузеров;
- артефакты отчета;
- trace на падениях.

Практика:

- добавить GitHub Actions workflow;
- запускать `npm test`;
- сохранять `playwright-report` при падении.

## 12. Финальный результат

К концу трека в проекте должны быть:

- `tests/e2e/smoke.spec.ts`;
- `tests/e2e/navigation.spec.ts`;
- `tests/e2e/theme.spec.ts`;
- `tests/e2e/contacts.spec.ts`;
- `tests/e2e/responsive.spec.ts`;
- README с командами запуска;
- понятная структура проекта;
- понимание, где нужен MCP, а где обычный Playwright.
