# QA Lab: CRUD API для дефектов

QA Lab — демонстрационная вкладка портфолио с полноценным CRUD-сценарием для сущности «дефект». Интерфейс находится на `/api-lab.html`, а JSON API доступен по маршрутам `/api/issues` и `/api/issues/{id}`.

Фича предназначена для демонстрации ручного тестирования, UI-автотестов и будущих API-тестов. Это публичная учебная среда без регистрации и без пользовательских аккаунтов.

## Быстрый старт

Требования:

- Node.js 22 LTS;
- npm;
- установленные браузеры Playwright для запуска E2E-тестов.

```bash
npm ci
npm run dev
```

После запуска:

- сайт и QA Lab: `http://localhost:8888/api-lab.html`;
- API: `http://localhost:8888/api/issues`;
- OpenAPI-контракт: [`docs/openapi.yaml`](./openapi.yaml);
- план будущих API-тестов: [`docs/api-test-plan.md`](./api-test-plan.md).

`npm run dev` запускает Netlify Dev, поэтому статический интерфейс, serverless-функция и локальное хранилище Netlify Blobs работают в одном окружении. Обычный Vite-сервер (`npm run serve:static`) подходит только для просмотра статической части: API-маршруты в нём недоступны.

## Модель данных

Дефект содержит следующие поля:

| Поле | Кто задаёт | Правила |
| --- | --- | --- |
| `id` | сервер | UUID, неизменяемое |
| `title` | клиент | строка, 3–120 символов после `trim` |
| `description` | клиент | строка, 10–2000 символов после `trim` |
| `severity` | клиент | `low`, `medium`, `high` или `critical` |
| `status` | клиент/сервер | `open`, `in_progress` или `resolved`; при создании по умолчанию `open` |
| `createdAt` | сервер | дата и время в ISO 8601, неизменяемое |
| `updatedAt` | сервер | дата и время в ISO 8601; меняется при `PATCH` |

Сервер отклоняет неизвестные поля и попытки передать `id`, `createdAt` или `updatedAt`. Заголовок и описание сохраняются уже без пробелов по краям.

## Изоляция демо-пространств

Каждый запрос обязан содержать заголовок:

```http
X-Demo-Workspace-Id: 3f64012e-4f50-4d6f-80ea-f51e840abc91
```

Значение должно быть UUID. Интерфейс создаёт UUID при первом открытии и сохраняет его в `localStorage` под ключом `qa-lab-workspace-id`. Благодаря этому данные одного браузерного профиля не смешиваются с данными другого.

Workspace ID — только ключ разделения данных, а не средство аутентификации. Тот, кто знает UUID, может обратиться к этому пространству. В QA Lab нельзя хранить персональные, конфиденциальные или иные чувствительные данные. Очистка хранилища браузера создаст новое пространство; старые записи перестанут быть видны в интерфейсе, но не удалятся немедленно на сервере.

## API

| Метод | Маршрут | Назначение | Успешный статус |
| --- | --- | --- | --- |
| `GET` | `/api/issues` | получить список, применить фильтры и поиск | `200` |
| `POST` | `/api/issues` | создать дефект | `201` |
| `GET` | `/api/issues/{id}` | получить один дефект | `200` |
| `PATCH` | `/api/issues/{id}` | частично изменить дефект | `200` |
| `DELETE` | `/api/issues/{id}` | удалить дефект | `204` |

Для `POST` и `PATCH` обязателен `Content-Type: application/json`. `PATCH` принимает только реально изменяемые поля и требует хотя бы одно поле.

### Список, поиск и фильтры

`GET /api/issues` поддерживает:

- `q` — поиск подстроки в `title` и `description` без учёта регистра, максимум 120 символов после `trim`;
- `severity` — одно из значений severity;
- `status` — одно из значений status.

Параметры можно комбинировать. Неизвестный или повторённый query-параметр приводит к `400 INVALID_QUERY`. Результат сортируется по `updatedAt` от новых изменений к старым, а при одинаковом времени — по `id` по возрастанию. `total` отражает число элементов после фильтрации.

### Формат ошибок

Все JSON-ошибки имеют единую форму:

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Request validation failed",
    "fields": {
      "title": "Field must contain at least 3 characters after trimming"
    },
    "requestId": "d2865864-225a-4026-a2d7-0c063a06f985"
  }
}
```

`fields` пуст для ошибок, не связанных с конкретным полем. Значение `requestId` также возвращается в заголовке `X-Request-Id` и помогает сопоставить ответ с серверным логом.

Все ответы запрещают кэширование через `Cache-Control: no-store` и содержат `Vary: X-Demo-Workspace-Id`. Для неподдерживаемого метода сервер отвечает `405` и добавляет `Allow` с допустимыми методами ресурса.

Полный перечень статусов и кодов ошибок зафиксирован в OpenAPI-контракте.

## Примеры запросов

В примерах используется постоянный UUID, чтобы несколько команд работали с одним пространством:

```bash
BASE_URL=http://localhost:8888
WORKSPACE_ID=3f64012e-4f50-4d6f-80ea-f51e840abc91
```

Создать дефект:

```bash
curl -i -X POST "$BASE_URL/api/issues" \
  -H "X-Demo-Workspace-Id: $WORKSPACE_ID" \
  -H "Content-Type: application/json" \
  --data '{
    "title": "Кнопка отправки неактивна",
    "description": "После заполнения обязательных полей кнопка остаётся неактивной.",
    "severity": "high"
  }'
```

Взять `id` из ответа и подставить его дальше:

```bash
ISSUE_ID=00000000-0000-0000-0000-000000000000
```

Получить список открытых дефектов с поиском:

```bash
curl -i \
  -H "X-Demo-Workspace-Id: $WORKSPACE_ID" \
  "$BASE_URL/api/issues?status=open&q=%D0%BA%D0%BD%D0%BE%D0%BF%D0%BA%D0%B0"
```

Получить один дефект:

```bash
curl -i \
  -H "X-Demo-Workspace-Id: $WORKSPACE_ID" \
  "$BASE_URL/api/issues/$ISSUE_ID"
```

Изменить только статус и severity:

```bash
curl -i -X PATCH "$BASE_URL/api/issues/$ISSUE_ID" \
  -H "X-Demo-Workspace-Id: $WORKSPACE_ID" \
  -H "Content-Type: application/json" \
  --data '{"status":"in_progress","severity":"critical"}'
```

Удалить дефект:

```bash
curl -i -X DELETE "$BASE_URL/api/issues/$ISSUE_ID" \
  -H "X-Demo-Workspace-Id: $WORKSPACE_ID"
```

## Хранение и конкурентные изменения

Данные хранятся в Netlify Blobs. Для каждого workspace используется один JSON-документ со списком дефектов:

- хранилище открывается с strong consistency;
- запись выполняется условно по ETag (`onlyIfNew` или `onlyIfMatch`);
- при конфликте сервер повторно читает актуальную версию и пробует запись до пяти раз;
- если устойчивый конфликт не разрешился, API отвечает `409 CONCURRENT_MODIFICATION`.

В production Netlify условные записи делают проверку лимита и CRUD-изменение
одного workspace атомарными относительно конкурентных запросов. В одном
пространстве разрешено не более 50 дефектов; 51-я последовательная попытка
создания получает `409 WORKSPACE_LIMIT_REACHED`.

Срок активности workspace — 30 дней после последней успешной мутации (`POST`, `PATCH` или `DELETE`). После этой даты API логически считает документ пустым. Чтение не продлевает срок. Физическое удаление старого Blob по расписанию сейчас не реализовано: следующая успешная запись в просроченное пространство перезапишет документ. Если потребуется гарантированное физическое удаление, нужно добавить отдельную scheduled function и индекс workspace с датами истечения.

## Локальное и production-окружение

В Netlify Dev Blobs работают локально. Эти данные не синхронизируются с production и могут исчезнуть при очистке локального состояния Netlify.

Локальный Blobs-сервер в `netlify-cli 26.2` не возвращает ETag при чтении и не
гарантирует атомарность нескольких одновременных conditional writes. Backend
восстанавливает стабильный ETag через локальный list/read/list fallback, поэтому
обычные последовательные CRUD-сценарии и лимит воспроизводимы. P2-тесты с
одновременными мутациями одного workspace следует запускать на Netlify deploy
preview: в локальном эмуляторе такие запросы могут все получить `201`, хотя
последующая запись затрёт часть параллельных изменений.

В production функция использует Blob store сайта Netlify. Переменные окружения или отдельная база данных для текущей реализации не нужны.

## Требования к деплою в Netlify

Репозиторий уже содержит необходимые настройки в `netlify.toml`:

- publish directory: `site`;
- functions directory: `netlify/functions`;
- bundler: `esbuild`;
- явные rewrites направляют `/api/issues` и `/api/issues/:id` в Function и
  сохраняют её исходные статусы, включая item-ответы `404` в Netlify Dev.

Для нового сайта в Netlify достаточно:

1. Подключить репозиторий.
2. Использовать Node.js 22.
3. Установить зависимости через `npm ci` (обычно это делает стандартный build pipeline Netlify).
4. Оставить publish directory и functions directory из `netlify.toml`.
5. После деплоя проверить `GET /.netlify/functions/issues` не требуется: публичный контракт использует только `/api/issues`.

Секреты для QA Lab не настраиваются. Перед публичной демонстрацией стоит помнить, что это открытое демо без авторизации и с ограничениями, описанными выше.
